# ADR-002: Sandbox Backend Abstraction

## Title

Sandbox Backend Abstraction for Command Execution

## Status

Accepted

## Context

IronClad needs to execute shell commands generated by the LLM. Direct execution on the host system would be dangerous because:

1. **Malicious commands** could damage the host system
2. **Unintended side effects** could affect user data
3. **Resource exhaustion** could crash the system
4. **Security vulnerabilities** could be exploited

Different execution environments are needed for different platforms and use cases:
- Docker containers for isolated, ephemeral execution
- WSL for Windows users without Docker
- Future: cloud-based sandboxes, VMs, etc.

## Decision

We implement a trait-based abstraction for sandbox execution:

```rust
#[async_trait]
pub trait SandboxBackend: Send + Sync {
    /// Execute a command in the sandbox
    async fn execute(&self, cmd: &str, workspace: &Path) -> Result<ExecutionResult>;
    
    /// Return the name of the backend
    fn name(&self) -> &str;
    
    /// Clone the backend into a new Box
    fn clone_box(&self) -> Box<dyn SandboxBackend>;
}
```

### Backend Implementations

#### Docker Backend
- Creates ephemeral containers for each command
- Mounts workspace directory as volume
- Enforces resource limits (CPU, memory)
- Network isolation configurable
- Container destroyed after execution

#### WSL Backend
- Uses Windows Subsystem for Linux
- Executes commands in specified distribution
- Workspace path translated to WSL path
- Lower isolation but faster for Windows users

### Factory Pattern

Backend creation is handled by a factory function:

```rust
pub async fn create_sandbox(config: &SandboxConfig) -> Result<Box<dyn SandboxBackend>> {
    match config.backend.as_str() {
        "docker" => Ok(Box::new(DockerBackend::connect(config)?)),
        "wsl" => Ok(Box::new(WslBackend::new(config.clone()))),
        other => Err(IronCladError::Config(format!("Unknown backend: {}", other))),
    }
}
```

## Consequences

### Positive
- Easy to add new backends (cloud, VM, etc.)
- Consistent interface across all backends
- Testable with mock implementations
- Configuration-driven backend selection

### Negative
- Trait object overhead (minimal)
- Backend-specific features harder to expose
- Error handling must be generic

## Alternatives Considered

### 1. Docker-Only
Only support Docker for execution. Rejected because it excludes Windows users without Docker and limits deployment options.

### 2. Direct Execution
Execute commands directly on host with user confirmation. Rejected as too dangerous for autonomous agent.

### 3. Remote Execution Only
All commands executed on remote servers. Rejected due to latency, cost, and privacy concerns.

### 4. Multiple Trait Objects
Different traits for different backend capabilities. Rejected as overly complex for current needs.

## Implementation Details

### Docker Backend Configuration

```toml
[sandbox]
backend = "docker"
socket = ""  # Auto-detect
default_image = "alpine:latest"
timeout_secs = 60
memory_limit_mb = 128
cpu_limit = 0.5
network_enabled = false
```

### WSL Backend Configuration

```toml
[sandbox]
backend = "wsl"
default_image = "Ubuntu"  # Distribution name
```

### Execution Result

```rust
pub struct ExecutionResult {
    pub exit_code: i32,
    pub stdout: String,
    pub stderr: String,
}
```

## Future Enhancements

1. **Cloud Sandboxes**: AWS Lambda, GCP Cloud Functions
2. **VM Backends**: Firecracker, gVisor
3. **Persistent Containers**: Reuse containers for session
4. **Resource Monitoring**: Track CPU/memory usage
5. **Execution Timeout**: Kill long-running commands
